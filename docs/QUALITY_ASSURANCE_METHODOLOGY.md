# 单智能体质量保障方法论

> 适用于 AI 辅助开发场景下的质量保障体系

---

## 一、测试核心哲学（强制）

### 1.1 五大原则

```
1. 测试的目标是验证业务逻辑正确性、发现生产环境问题
   ——绝不单纯追求代码覆盖率

2. 必须覆盖正向、负向、边界、异常、依赖五种场景
   ——五场景缺一不可

3. 断言必须验证具体业务逻辑
   ——禁止仅验证 status 的弱断言

4. 无需求文档时，先基于代码梳理业务规则和风险点
   ——再围绕规则设计测试

5. 测试代码只允许修改 tests/ 下文件
   ——禁止修改 src/ 等业务代码
```

### 1.2 五场景定义

| 场景 | 定义 | 示例 |
|------|------|------|
| **正向** | 正常流程，输入合法数据 | 正常审批通过 |
| **负向** | 错误输入，验证系统拒绝 | 审批不存在的记录 |
| **边界** | 临界值、极限情况 | 过期审批清理 |
| **异常** | 系统故障、网络错误 | 数据库连接失败 |
| **依赖** | 第三方服务、外部接口 | Mock 外部 API |

## 二、多角色自我审查法

### 2.1 角色切换模型

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  开发视角 │ →  │  测试视角 │ →  │  架构视角 │ →  │  安全视角 │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
                        │
                        ▼
              ┌───────────────────┐
              │ 技术委员会视角     │
              │ （最终审核）       │
              └───────────────────┘
```

### 2.2 各视角审查清单

#### 开发视角
- [ ] 代码实现是否符合需求？
- [ ] 变量命名是否清晰？
- [ ] 函数职责是否单一？
- [ ] 是否有重复代码？

#### 测试视角
- [ ] 测试覆盖了所有业务规则吗？
- [ ] 五场景都覆盖了吗？
- [ ] 断言是强断言吗？
- [ ] 如果故意破坏代码，测试会失败吗？

#### 架构视角
- [ ] 设计是否符合分层架构？
- [ ] 模块间耦合是否合理？
- [ ] 扩展性是否满足需求？
- [ ] 是否有性能瓶颈？

#### 安全视角
- [ ] 输入是否验证？
- [ ] 是否有注入风险？
- [ ] 敏感数据是否保护？
- [ ] 权限控制是否正确？

#### 技术委员会视角
- [ ] 业务逻辑是否正确？
- [ ] 测试是否真的有效？
- [ ] 风险是否可控？
- [ ] 是否可以发布？

## 三、测试分层模型

### 3.1 四层测试模型

```
┌─────────────────────────────────┐
│  L4 - 端到端测试                 │  业务闭环正确性
│  （E2E / 业务规则验证）          │
├─────────────────────────────────┤
│  L3 - 集成测试                   │  模块间协作正确性
│  （接口契约 / 流程验证）         │
├─────────────────────────────────┤
│  L2 - 组件测试                   │  类/模块级行为正确性
│  （组件行为 / 状态机）           │
├─────────────────────────────────┤
│  L1 - 单元测试                   │  函数级逻辑正确性
│  （函数 / 方法验证）             │
└─────────────────────────────────┘
```

### 3.2 各层有效性标准

| 层级 | 验证目标 | 有效性标准 |
|------|----------|------------|
| L1 单元 | 函数逻辑正确 | 变异 kill rate ≥ 80% |
| L2 组件 | 组件行为正确 | 组件状态变异后测试能检测 |
| L3 集成 | 模块协作正确 | 接口契约破坏后测试能检测 |
| L4 端到端 | 业务闭环正确 | 业务规则破坏后测试能检测 |

## 四、变异测试验证法

### 4.1 核心思想

```
测试有效吗？
    ↓
故意破坏代码 → 运行测试 → 测试应该失败
    ↓
如果测试失败 → 测试有效 ✅
如果测试通过 → 测试无效 ❌
```

### 4.2 变异验证门流程

1. **备份**：备份源文件
2. **变异**：故意引入已知 bug
3. **验证**：运行相关测试
4. **断言**：测试必须失败（否则测试无效）
5. **恢复**：恢复源文件

### 4.3 已知bug清单

每个项目维护一个"已知 bug 清单"，用于：
- 验证测试体系有效性
- 新员工培训
- 回归测试基准

## 五、业务规则驱动测试

### 5.1 核心思想

```
不是围绕代码写测试
而是围绕业务规则写测试
```

### 5.2 业务规则格式

```
### BR-XX: 规则名称

- **代码依据**: 文件名
- **位置**: 第X行
- **说明**: 详细描述业务规则
- **正向测试**: ✅ 正常情况
- **负向测试**: ✅ 异常情况
```

### 5.3 工作流

```
梳理代码 → 提取业务规则 → 围绕规则写测试 → 每条规则正向+负向
```

## 六、CI 守卫机制

### 6.1 检测反模式

| 反模式 | 说明 | 危害 |
|--------|------|------|
| 弱断言 | `assert hasattr(obj, 'attr')` | 只验证存在，不验证行为 |
| 失败跳过 | `pytest.skip()` | 逃避问题，不是解决问题 |
| 异常吞没 | `except Exception: pass` | 隐藏错误，导致问题积累 |

### 6.2 集成位置

- **pre-commit hook**：提交前检查
- **CI 流水线**：合并前检查
- **IDE 插件**：编码时提醒

## 七、标准工作流（六步法）

```
Step 1: 理解需求 → 梳理业务规则
   ↓
Step 2: 写测试 → 先写失败测试（测试先行）
   ↓
Step 3: 写代码 → 让测试通过
   ↓
Step 4: 自我审查 → 切换视角审查代码
   ↓
Step 5: 验证测试 → 运行变异验证门
   ↓
Step 6: 记录问题 → 更新 BUG_REGISTRY
```

## 八、BUG 登记机制

### 8.1 登记要素

每个 BUG 必须包含：
- 问题描述 + 严重级别
- 根因分析
- 复现步骤
- 修复方案
- 验证结果
- 改进措施（防止再次发生）

### 8.2 严重级别

| 级别 | 定义 | 修复时限 |
|------|------|----------|
| P0 | 阻断性，系统不可用 | 立即 |
| P1 | 核心功能不可用 | 本版本 |
| P2 | 非核心功能问题 | 下一版本 |
| P3 | 微小问题，可延后 | 后续版本 |

## 九、质量保障公式

```
质量 = 业务规则驱动
     + 测试先行
     + 变异验证
     + 分层有效性
     + 自我审查清单
     + 自动化工具门禁
```

# 技术委员会运作指南

> 单智能体场景下的技术委员会运作机制

---

## 一、技术委员会职责

### 1.1 核心职责

| 职责 | 说明 | 频率 |
|------|------|------|
| **代码审查** | 审核核心代码变更质量 | 每变更 |
| **风险评估** | 识别技术风险和债务 | 每迭代 |
| **测试审核** | 审核测试有效性和覆盖率 | 每版本 |
| **阶段验收** | 验收阶段目标完成情况 | 每阶段 |
| **技术决策** | 重大技术方案决策 | 按需 |

### 1.2 权力边界

- ✅ 可以：要求修复问题、暂停发布、调整优先级
- ✅ 可以：提出改进建议、制定技术规范
- ❌ 不可以：修改业务代码（只能提出问题）
- ❌ 不可以：绕过测试流程

## 二、三会议一报告机制

### 2.1 代码审查会

**频率**: 每周 / 每重大变更

**议程**:
1. 本周变更回顾
2. 核心代码审查
3. 问题登记和跟踪
4. 下周重点关注

**产出**:
- 审查意见清单
- 问题登记（BUG_REGISTRY）
- 改进建议

### 2.2 风险评估会

**频率**: 每迭代

**议程**:
1. 当前技术债务盘点
2. 风险识别和分级
3. 优先级排序
4. 应对方案制定

**产出**:
- 风险清单
- 优先级排序
- 应对计划

### 2.3 阶段验收会

**频率**: 每阶段末

**议程**:
1. 阶段目标回顾
2. 问题修复验证
3. 测试有效性验证
4. 验收结论

**产出**:
- 验收报告
- 进入下一阶段确认
- 遗留问题清单

### 2.4 技术委员会月报

**频率**: 每月

**内容**:
1. 本月技术健康度
2. 问题统计和趋势
3. 测试有效性指标
4. 下月重点计划

## 三、问题生命周期管理

### 3.1 问题状态流转

```
发现 → 定级 → 指派 → 修复 → 验证 → 关闭 → 复盘
  │      │      │      │      │      │      │
  ▼      ▼      ▼      ▼      ▼      ▼      ▼
  TC    主席   负责人   开发   测试    TC    全员
登记   优先级  排期    编码   验证    关闭  总结
```

### 3.2 问题登记表单

每个问题必须包含：

```
## BUG-XXX: 问题标题

- **严重级别**: P0 / P1 / P2 / P3
- **状态**: 新建 / 修复中 / 已修复 / 已关闭
- **文件路径**: src/xxx/yyy.py
- **错误描述**: 详细描述问题现象
- **根因分析**: 为什么会出现这个问题
- **复现步骤**: 如何复现这个问题
- **修复方案**: 如何修复
- **验证结果**: 如何验证修复有效
- **改进措施**: 如何防止再次发生
```

### 3.3 严重级别定义

| 级别 | 定义 | 修复时限 | 示例 |
|------|------|----------|------|
| **P0** | 阻断性，系统不可用 | 立即 | 主流程崩溃、数据丢失 |
| **P1** | 核心功能不可用 | 本版本 | 治理闭环断裂、核心测试失效 |
| **P2** | 非核心功能问题 | 下一版本 | 代码质量、硬编码、调试语句 |
| **P3** | 微小问题，可延后 | 后续版本 | 注释、命名、小优化 |

## 四、四眼原则（变更审查）

### 4.1 变更分级审查

| 变更级别 | 审查要求 | 示例 |
|----------|----------|------|
| **P0 紧急变更** | 技术委员会主席 + 1名委员 双审 | 线上故障修复 |
| **P1 重要变更** | 2名委员 双审 | 核心功能修改 |
| **P2 一般变更** | 1名委员 审查 | 非核心功能优化 |
| **P3 微小变更** | 自审 + CI通过 | 文档、注释修改 |

### 4.2 审查清单

**代码审查清单**:
- [ ] 业务逻辑是否正确？
- [ ] 边界条件是否处理？
- [ ] 异常情况是否处理？
- [ ] 是否有硬编码？
- [ ] 是否有调试语句残留？
- [ ] 命名是否清晰？
- [ ] 是否有重复代码？

**测试审查清单**:
- [ ] 测试验证的是业务逻辑吗？
- [ ] 断言是强断言吗？
- [ ] 五场景都覆盖了吗？
- [ ] 如果故意破坏代码，测试会失败吗？
- [ ] 测试是否与业务规则对应？

**架构审查清单**:
- [ ] 设计是否符合分层架构？
- [ ] 模块间耦合是否合理？
- [ ] 扩展性是否满足需求？
- [ ] 是否有性能瓶颈？
- [ ] 安全性是否考虑？

## 五、阶段收敛模型

### 5.1 三阶段递进

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   P0 阶段    │───▶│   P1 阶段    │───▶│   P2 阶段    │
│  生存底线    │    │  质量达标    │    │  持续优化    │
└─────────────┘    └─────────────┘    └─────────────┘
```

### 5.2 各阶段验收标准

#### P0阶段收敛标准（生存底线）
- [ ] 所有P0问题修复并验证
- [ ] 核心测试通过（单元+集成）
- [ ] 无阻断性bug
- [ ] 测试体系不退化

#### P1阶段收敛标准（质量达标）
- [ ] 所有P1问题修复并验证
- [ ] 变异测试 kill rate ≥ 80%
- [ ] 核心业务规则测试覆盖100%
- [ ] 五场景覆盖达标

#### P2阶段收敛标准（持续优化）
- [ ] P2问题按计划修复（可选）
- [ ] 技术债务按计划偿还（可选）
- [ ] 建立持续改进机制

### 5.3 阶段验收清单

进入下一阶段前必须确认：

- [ ] **问题清单收敛**: 当前级别问题全部关闭或降级
- [ ] **测试验证通过**: 全量测试 + 变异验证门通过
- [ ] **文档更新完成**: BUG_REGISTRY、业务规则清单更新
- [ ] **经验教训总结**: 根因分析、改进措施已记录
- [ ] **风险评估完成**: 已知风险已识别并制定预案
- [ ] **技术委员会签字**: 主席签字确认阶段验收通过

## 七、单智能体运作模式

### 7.1 角色切换法

```
开发视角 → 测试视角 → 架构视角 → 安全视角 → 技术委员会视角
```

**操作要点**:
1. 每个角色独立思考，不跳过
2. 每个角色的审查意见都记录
3. 技术委员会视角做最终决策

### 7.2 自我审查清单

每次完成编码后，必须回答以下问题：

**业务逻辑**:
- [ ] 代码实现的业务逻辑是否正确？
- [ ] 边界条件是否处理？
- [ ] 异常情况是否处理？

**测试质量**:
- [ ] 测试验证的是业务逻辑，而不是实现细节？
- [ ] 断言是强断言，不是仅验证 status？
- [ ] 覆盖了正向、负向、边界、异常、依赖五种场景？

**测试有效性**:
- [ ] 如果故意破坏这段代码，测试会失败吗？
- [ ] 变异测试能杀死这个变异吗？
- [ ] 有没有测试盲区？

**代码质量**:
- [ ] 有没有硬编码？
- [ ] 有没有复制粘贴错误？
- [ ] 有没有调试语句残留？

### 7.3 记录留痕

- 所有问题都记录到 BUG_REGISTRY.md
- 所有决策都有理由说明
- 所有验证都有测试证据
- 所有阶段验收都有签字

## 八、质量健康度指标

### 8.1 核心指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 全量测试通过率 | ≥ 99% | 不包括已知跳过 |
| 变异 kill rate | ≥ 80% | 核心模块 |
| P0问题修复时长 | ≤ 24小时 | 从发现到关闭 |
| P1问题修复时长 | ≤ 1迭代 | 从发现到关闭 |
| 业务规则覆盖率 | 100% | 核心规则 |
| 五场景覆盖率 | ≥ 90% | 核心功能 |

### 8.2 趋势监控

每月跟踪以下趋势：
- 问题数量趋势（新增 vs 关闭）
- 测试有效性趋势（kill rate 变化）
- 技术债务趋势（P2/P3问题数量）
- 修复时长趋势（MTTR）

## 九、持续改进机制

### 9.1 复盘机制

1. 问题是怎么发现的？
2. 为什么测试没有发现？
3. 根因是什么？
4. 如何防止再次发生？

### 9.2 定期回顾

每月回顾：
1. 本月发现了多少问题？
2. 哪些类型的问题最多？
3. 测试有效性是否下降？
4. 技术债务是增加还是减少？
5. 下月重点改进什么？

## 十一、CI/CD 集成与自动化验证

### 11.1 CI 流水线结构

```
┌─────────────────────────────────────────────────────────────────┐
│                      CI Pipeline                                │
├─────────────────────────────────────────────────────────────────┤
│  push/pr → Lint → Unit → Component → Integration →             │
│                     Governance → Effectiveness Gate →           │
│                     Security Scan → Quality Gate →              │
│                     Mutation Test (main) → Full Test            │
└─────────────────────────────────────────────────────────────────┘
```

### 11.2 质量门禁配置

| 门禁 | 阈值 | 位置 |
|------|------|------|
| CI Guard | 无反模式 | pre-commit / CI |
| 单元测试 | 100%通过 | CI |
| 集成测试 | 100%通过 | CI |
| 变异验证门 | 100%通过 | CI |
| 变异测试 | Kill Rate ≥ 80% | CI (main分支) |
| 安全扫描 | 无严重漏洞 | CI |

### 11.3 自动化验证流程

1. **提交阶段**: pre-commit hooks 自动运行
2. **PR阶段**: 全量测试 + 有效性门控 + 安全扫描
3. **合并阶段**: mutmut 全量变异测试 (仅main分支)
4. **发布阶段**: 质量门禁通过才能部署

### 11.4 CI 产出物

| 产出物 | 描述 | 保留时间 |
|--------|------|----------|
| 测试报告 | pytest HTML报告 | 7天 |
| 覆盖率报告 | coverage HTML报告 | 7天 |
| 变异报告 | mutmut HTML报告 | 7天 |
| 安全报告 | bandit + pip-audit | 7天 |

## 十三、变异测试结果跟踪

### 13.1 变异测试指标

| 指标 | 目标值 | 当前值 |
|------|--------|--------|
| 变异体数量 | - | 待执行 |
| 杀死数量 | - | 待执行 |
| 幸存数量 | - | 待执行 |
| Kill Rate | ≥ 80% | 待执行 |

### 13.2 变异测试执行计划

1. **短期**: 使用自定义脚本执行变异测试，记录基线
2. **中期**: 在CI中集成自定义变异测试，每周运行
3. **长期**: 迁移到Linux环境，运行mutmut全量测试

### 13.3 幸存变异体处理流程

1. 分析幸存变异体对应的代码
2. 确定是否需要添加测试用例
3. 编写测试用例覆盖该变异体
4. 重新运行变异测试验证

## 十五、持续改进流程

### 15.1 改进提案流程

1. **提出**: 任何成员可以提出改进建议
2. **评估**: 技术委员会评估可行性和优先级
3. **立项**: 批准后纳入改进计划
4. **实施**: 负责人实施改进
5. **验证**: 测试验证改进效果
6. **复盘**: 总结经验教训

### 15.2 改进优先级

| 优先级 | 标准 | 示例 |
|--------|------|------|
| P0 | 影响核心功能 | 测试体系失效 |
| P1 | 影响开发效率 | CI流水线优化 |
| P2 | 影响代码质量 | 技术债务清理 |
| P3 | 提升体验 | 文档完善 |

### 15.3 改进跟踪

所有改进项记录到 BUG_REGISTRY.md 的"待处理事项"表格中，跟踪状态：

| 状态 | 说明 |
|------|------|
| 待评估 | 等待技术委员会评估 |
| 已批准 | 已批准，等待实施 |
| 实施中 | 正在实施 |
| 已完成 | 已完成并验证 |
